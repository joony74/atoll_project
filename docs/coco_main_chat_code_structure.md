# 코코 메인챗 코드 구조 설계

## 1. 목적

이 문서는 코코앱 메인챗을 실제 코드 단위로 어떻게 분리할지 정의하기 위한 구조 설계 문서다.

목표는 다음과 같다.

- `app.py`에 몰려 있는 메인챗 관련 책임을 줄인다.
- 메인챗과 학습채팅의 상태 관리를 분리 가능하게 만든다.
- 이후 검색 기반 응답, 수학 핸드오프, 응답 조립기를 붙이기 쉬운 구조를 만든다.

## 2. 분리 원칙

처음부터 너무 많은 파일로 쪼개기보다, 현재 복잡도가 가장 높은 부분부터 먼저 분리한다.

첫 단계에서 우선 고정할 축은 다음 네 개다.

- `contracts`
- `state`
- `main_engine`
- `ui`

그리고 메인챗 응답이 조금 더 살아나기 시작하는 시점에는, 응답 오케스트레이션을 얇게 유지하기 위해 다음 파일들을 바로 붙인다.

- `router`
- `composer`
- `handoff`
- `grounding`
- `followup`

## 3. 권장 패키지 구조

현재 기준 권장 패키지 구조는 다음과 같이 가져간다.

- `app/chat/__init__.py`
- `app/chat/contracts.py`
- `app/chat/state.py`
- `app/chat/router.py`
- `app/chat/main_engine.py`
- `app/chat/composer.py`
- `app/chat/handoff.py`
- `app/chat/grounding.py`
- `app/chat/followup.py`
- `app/chat/ui.py`

## 4. 파일별 책임

### 4-1. `app/chat/contracts.py`

메인챗과 학습채팅에서 공통으로 쓰는 자료 구조를 정의한다.

- 메시지 타입
- 문서 요약 타입
- 앱 상태 타입

초기에는 `TypedDict` 수준으로 시작하고, 구조가 더 안정되면 dataclass나 pydantic 모델로 확장할 수 있다.

### 4-2. `app/chat/state.py`

상태와 저장소 관련 책임을 모은다.

- 기본 상태 생성
- 상태 마이그레이션
- 상태 로드/저장
- 문서 로드/저장/삭제
- 최근 타겟 복원
- 현재 활성 채팅 모드 결정
- 메인챗/학습채팅 메시지 append
- 문서 우선순위 승격

즉, 상태와 persistence의 중심 파일이다.

### 4-3. `app/chat/main_engine.py`

메인챗 응답 오케스트레이션 진입점이다.

- 입력을 받아 어떤 응답 흐름을 탈지 결정
- `router`, `composer`, `handoff`를 연결
- 메인챗의 얇은 조정 레이어 역할 유지

이 파일에는 답변 문구를 길게 쌓지 않고, 각 하위 모듈을 연결하는 책임만 남기는 것이 목표다.

### 4-4. `app/chat/router.py`

메인챗 입력의 의도를 가볍게 분류한다.

- 스몰톡
- 앱 사용 질문
- 수학 질문
- 일반 질문

아직은 룰 기반 분류로 시작하고, 나중에 더 정교한 판단기로 확장할 수 있다.

### 4-5. `app/chat/composer.py`

자연스러운 메인챗 응답 문장을 조립한다.

- 스몰톡 답변
- 앱 안내 답변
- 일반 질문 답변
- 불확실성 말투 규칙

즉, 코코의 메인챗 말투를 관리하는 레이어다.

### 4-6. `app/chat/handoff.py`

학습채팅으로 넘기는 문장을 관리한다.

- 수학 질문 감지 후 학습리스트 안내
- 문서가 있을 때와 없을 때의 핸드오프 문장 분리
- 너무 차갑지 않게 전환하는 연결 문구 관리

### 4-7. `app/chat/grounding.py`

메인챗이 내부적으로 참고할 로컬 근거를 만든다.

- 현재 활성 채팅 모드
- 저장된 학습리스트 개수
- 최근 선택된 학습리스트 이름
- 앱 상태 질문에 대한 로컬 사실 요약

즉, 화면에는 드러나지 않지만 답변이 덜 흔들리게 만드는 내부 근거 레이어다.

### 4-8. `app/chat/followup.py`

직전 assistant 질문과 현재 짧은 사용자 답을 이어받는 레이어다.

- `그래`, `응`, `좋아` 같은 짧은 답 감지
- 직전 assistant 질문 문맥 읽기
- 꼬리질문, 역질문 뒤의 짧은 응답을 자연스럽게 이어가기

즉, 메인챗이 한 턴 단위가 아니라 흐름 단위로 대답하게 만드는 보조 레이어다.

### 4-9. `app/chat/ui.py`

메인챗 렌더링 보조 기능을 모은다.

- 프롬프트 제출 큐잉
- 대화 HTML 렌더
- 최근 메시지 위치 스크롤
- 빈 상태 렌더 기준

즉, Streamlit 화면과 직접 맞닿는 보조 UI 레이어다.

## 5. `app.py`에 남길 책임

1차 분리 후에도 `app.py`는 완전히 비워지지 않는다.

남겨둘 책임은 다음과 같다.

- Streamlit 페이지 설정
- 업로드 이벤트 처리
- 사이드바 화면 구성
- 체크패널 렌더링
- 수학 풀이 파이프라인 호출
- 전체 화면 CSS 주입

즉, `app.py`는 조립과 레이아웃 중심 파일로 남기고, 메인챗 로직은 `app/chat` 쪽으로 밀어낸다.

## 6. 호출 흐름

현재 구조 기준 호출 흐름은 다음과 같다.

1. `app.py`가 상태를 불러온다.
2. 상태 관련 처리는 `app.chat.state`가 담당한다.
3. 메인챗 입력이 들어오면 `app.chat.main_engine`이 진입점이 된다.
4. `main_engine`은 `router`로 의도를 나눈다.
5. `main_engine`은 필요할 때 `grounding`으로 앱 상태와 학습리스트 상태를 읽는다.
6. 직전 assistant 질문과 현재 짧은 답이 연결될 때는 `followup`이 먼저 흐름을 이어받는다.
7. 일반 응답은 `composer`, 수학 전환 문장은 `handoff`가 만든다.
8. 대화 렌더링과 스크롤 보정은 `app.chat.ui`가 담당한다.
9. `app.py`는 최종 결과를 화면에 조립한다.

## 7. 다음 확장 구조

다음 단계에서는 `grounding`을 지금의 로컬 근거 수준에서 더 확장한다.

- 내부 검색 후보 정리
- 검증 규칙 추가
- 이후 외부 검색 연계 전 준비
- followup과 grounding이 함께 작동하도록 문맥 처리 확장

## 8. 구현 순서

### 1단계

- `contracts.py` 추가
- `state.py` 추가
- `main_engine.py` 추가
- `ui.py` 추가

### 2단계

- `app.py`에서 해당 함수들을 새 모듈로 이동
- import 연결 정리
- 기존 동작 유지 확인

### 3단계

- `router.py` 추가
- `composer.py` 추가
- `handoff.py` 추가
- 메인챗 응답을 라우팅 구조로 연결

### 4단계

- `grounding.py`를 로컬 근거 레이어에서 검색/검증 레이어로 확장
- `followup.py`를 붙여 꼬리질문과 짧은 응답의 문맥 연결 강화

## 9. 핵심 원칙

현재 단계의 핵심은 “기능을 많이 넣는 것”이 아니라 “기능이 들어갈 자리를 먼저 만드는 것”이다.

즉,

- 상태는 `state.py`
- 메인챗 오케스트레이션은 `main_engine.py`
- 메인챗 말투는 `composer.py`
- 수학 전환 문장은 `handoff.py`
- 로컬 근거는 `grounding.py`
- 꼬리질문 연결은 `followup.py`
- 화면 보조는 `ui.py`
- 자료 구조는 `contracts.py`
- 입력 분류는 `router.py`

이 원칙을 먼저 고정하는 것이 중요하다.
